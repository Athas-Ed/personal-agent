# 作品集要点与加分项：把“个人工作台 Agent”做成求职作品

> 目标：让面试官在 3–5 分钟内看懂“你做了什么、为什么这样做、怎么验证它靠谱”，并且能**一键跑起来**。

---

## 作品集发布前必须注意的要点（很关键）

### 1) 安全与合规（第一优先级）

- **绝对不要提交密钥/个人信息**
  - 检查 `.env` 是否被提交（你仓库里当前存在 `.env` 文件：作品集发布前建议删除并只保留 `.env.example`）。
  - 检查 `data/`、`.cache/`、`study_files/` 中是否包含私人内容（聊天记录、真实姓名、邮箱、学校信息等）。
- **对外展示的第三方能力要有降级**
  - 你已做了：无 `AMAP_KEY` 时天气插件不加载；RAG 失败降级纯对话。
  - 作品集里要明确写出“缺少 Key 会怎样、如何关闭/开启”。
- **许可证与引用**
  - 建议添加 `LICENSE`（MIT/Apache-2.0 任选其一，面试官会更放心）。

### 2) 可复现运行（面试官要能跑）

- **提供“最小可运行路径”**：
  - 不要求向量库/不要求 MCP/不要求高德 Key，也能跑：至少能打开 Streamlit + 对话（纯对话）。
  - 在 README 里给出“5 分钟跑起来”的命令。
- **提供“验收清单”**：
  - 例如：启动成功、能对话、能上传文件入库、能查询 RAG、能导出学习笔记（写入 `study_files/`）、能显示工具调用轨迹。

### 3) 工程质量信号（会直接影响评价）

- **类型与静态检查**：哪怕只做到核心模块，也能显著加分（mypy/pyright）。
- **格式化与 lint**：ruff + black（或只用 ruff format）。
- **测试**：不是越多越好，而是“抓关键风险点”
  - 例如：路径安全（`resolve_under`）、工具写入确认（`confirm_write`）、RAG 降级策略、MCP 工具加载失败不影响主流程。
- **日志与可观测**：
  - 至少要有统一 logger（标准库 logging），关键链路记录：模型调用、工具调用、异常降级路径。

### 4) 面试展示材料（比“多写功能”更重要）

- **截图/短视频 Demo**：比长 README 更有效（30–90 秒最佳）。
- **架构图**：1 张图说明 Streamlit → Agent → Tools/MCP → RAG/Vectorstore。
- **明确你解决了哪些“真实工程坑”**
  - 你已经做的点就很加分：证书/代理兜底、Streamlit 勾选性能优化、文件写入确认与安全路径、RAG 降级。

## 建议新增/增强的“加分项”（按投入产出比排序）

### A 级（强烈建议，成本低、加分明显）

1) **一键本地启动的“开发者体验”再强化**
   - 增加 `run.bat --check` 的输出更明确（你已经有 `--check`，可以在 README 里把它当作“自检”能力宣传）。
   - README 增加“常见失败排查”：证书、代理、Key、端口占用。

2) **配置与模式切换更清晰**
   - 增加 `APP_MODE`：
     - `chat_only`（不走向量库）
     - `rag`（启用向量库）
     - `tools_agent`（启用工具调用）
   - 面试时可以说：我把功能拆成“可组合模式”，便于逐步启用、便于排障。

3) **把“工具调用轨迹”做成可下载/可回放**
   - 你已经在 UI 展示 tool events。
   - 建议增加“导出轨迹 JSON”按钮：面试官喜欢“可复盘”的工程设计。

4) **把关键链路做成 CLI**
   - 例如：`python -m scripts.smoke_tool_calling_chain`（你已有 `scripts/smoke_tool_calling_chain.py`）
   - 让面试官不打开 UI 也能验证核心能力。

### B 级（中等成本，但很容易形成亮点）

5) **引入 LangSmith / OpenTelemetry 的 tracing（可选）**
   - 亮点：你不仅能跑，还能观测 agent 调用链、工具耗时、失败点。
   - 注意：作品集里要支持关闭（避免需要账号/联网）。

6) **RAG 质量增强：加入重排（rerank）或至少加“引用质量”**
   - 在 UI 里展示：
     - 每个引用 chunk 的来源（你已有 source）
     - chunk 相似度/排序理由（可选）
   - 让回答更“可解释”，面试官会觉得你理解 RAG 工程。

7) **Prompt 版本化与评测（evaluation）**
   - 建一个 `prompts/` 目录，给系统提示词/模板做版本号。
   - 写一个最小评测脚本：
     - 固定 10 个问题
     - 统计：是否触发工具、是否降级、是否引用、耗时
   - 这是“工程化 LLM 应用”的典型加分项。

8) **更严格的“写入安全”**
   - 你已经用 `confirm_write` 防误写。
   - 可以再加一层：写入前先返回“计划”，用户点确认后再真正写（UI 两段式）。

### C 级（成本更高，适合冲刺亮点）

9) **容器化（Docker）+ 无状态演示**
   - 对 Python/Windows 项目并非必需，但如果你想投偏后端/平台岗会加分。
   - 关键是：容器内仍要处理证书/代理/密钥注入（用 env）。

10) **把 MCP 做成“多 server + 动态配置”**
   - 目前 `MCPClientService` 只连本地一个 server。
   - 作品集亮点可以是：
     - `mcp_servers.yaml`（或 json）配置多 server
     - UI 里展示可用工具列表并允许开关

11) **权限模型（工具白名单/黑名单）**
   - 面试官会问：模型能不能乱写文件？
   - 你可以说：我有 scope + confirm_write + 白名单工具 + 默认只读。

## 你这个项目“已经具备的亮点”，建议在作品集里突出

- **LCEL/Runnable 思维**：`message_chain`、`tool_calling_loop` 组合成链路（不是把逻辑写死在 UI 里）。
- **工具注册表模式**：本地工具 + MCP 工具统一装配、可选加载、失败隔离。
- **可插拔插件**：天气插件不污染 core。
- **RAG 降级**：知识库不可用时仍能对话，保证体验。
- **Windows/代理/证书治理**：非常贴近真实企业环境（比“做个demo”更像工程）。
- **Streamlit 性能坑处理**：从大量 checkbox 改为 form + multiselect，属于真实工程优化。

## 推荐的作品集呈现方式（让面试官快速理解）

### README 结构建议

- **一句话**：这是什么（个人学习工作台：RAG + 工具调用 + 笔记导出）
- **核心特性**：3–6 条（带截图）
- **Quickstart**：最短命令（含 `--check` 自检）
- **功能演示**：GIF/短视频链接（或本地录屏说明）
- **架构图**：链路图 + 关键模块表
- **技术要点**：列出你解决的工程问题（证书/代理、降级、工具安全等）
- **Roadmap**：未来可扩展点（只写 5 条以内）

### 面试讲解话术（30 秒版本）

- 我做了一个 Streamlit 工作台，把 DeepSeek(OpenAI兼容) 接到 LangChain；  
  核心是一个 LCEL 风格的 tool-calling loop：模型能自动选工具（本地工具 + MCP 工具）完成任务；  
  同时支持 Chroma 的本地向量库做 RAG，并且在检索失败时自动降级为纯对话；  
  项目针对 Windows/代理/证书环境做了兜底，并对文件写入做了 scope/confirm 约束，保证安全可控。
