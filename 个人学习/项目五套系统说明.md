## 项目里的“五套系统”：scripts / tools / utils / skills / services

你现在看到的四块东西，其实对应的是四种完全不同的“职责层次”。把它们区分清楚，项目结构就不会再歧义，也更容易维护依赖方向。

---

## 1) `scripts/`：命令行入口（可执行脚本层）

### 定位
`scripts/` 用来放**可以直接运行的脚本入口**（CLI wrappers）。它们的价值在于：
- 把“如何运行”标准化（参数、退出码、stdout/stderr）
- 让 `run.bat`、CI、手动运维有稳定调用点

### 放什么
- 只放“入口薄壳”：参数解析、调用 `src/*`、打印结果、返回退出码
- 典型例子：`scripts/sync_skills.py`、`scripts/prepare_ca_bundle.py`、`scripts/env_to_cmd.py`

### 不放什么（重点约束）
- 不要在 `scripts/` 堆业务逻辑/可复用逻辑  
  逻辑应该下沉到 `src/services/` 或 `src/utils/`（或更合适的 `src/tools/`）

### 依赖方向（关键）
**允许**：`scripts/` → `src/*`  
**禁止**：`src/*` → `scripts/`

---

## 2) `src/tools/`：运行时“工具模块”（被 import 的工具层）

> 这里的 “tools” 指的是：被核心逻辑/服务层调用的“工具能力模块”，不是“脚本工具箱”。

### 定位
`src/tools/` 是项目运行时的一部分，代码会被 `import`，通常具备：
- 明确的可复用 API（函数/类）
- 更偏“能力封装”，例如文件系统安全读写、对外工具加载等

### 放什么
- 可被其他模块稳定复用的工具封装
- 通常会被 `src/services/*` 或 `src/core/*` 引用

### 重点
如果某段代码：
- 需要被多处 import
- 需要单测/类型约束
- 需要明确 API 边界

那它更适合放在 `src/tools/`（或更轻量的 `src/utils/`），而不是顶层脚本目录。

---

## 3) `src/utils/`：通用辅助模块（小封装/纯函数层）

### 定位
`src/utils/` 放的是**跨模块复用的通用代码**，通常具备：
- 尽量“无业务语义”或“业务无关”
- 依赖轻（尽量不依赖服务层、外部网络、复杂状态）
- 更偏纯函数/小封装，方便复用与测试

### 放什么
- 轻量工具函数与小模块：字符串/路径/时间处理、格式化、简单解析器、轻量配置处理等
- “可复用但不值得做成 service 的能力”：例如证书 PEM 提取、`.env -> cmd set` 生成、技能 registry 构建等

### 不放什么（边界）
- 不放“需要连接复用/缓存/重试/外部集成”的流程（那是 `src/services/`）
- 不放“需要非常明确领域 API / 更像能力模块”的东西（更适合 `src/tools/`）

### 与 `src/tools/` 怎么区分（实用规则）
- **更像库函数/小模块**：放 `src/utils/`
- **更像一组稳定能力 API（被当作“工具能力”使用）**：放 `src/tools/`
- 不确定时优先放 `src/utils/`，等出现明显的 API 边界/扩展需求再提升到 `src/tools/`

---

## 4) `skills/`：技能资产（内容/元数据/脚本的“产品层”）

### 定位
`skills/` 是“资产仓库”，放的是你定义的技能内容，它更像**插件/资源包**，而不是代码分层里的某个模块。

在这个项目里，`skills/` 的核心是：
- 每个 skill 目录的 `SKILL.md`（frontmatter 元数据）
- 可选的 `scripts/*.py`（技能自己的脚本）
- `skills/registry.json`（索引/注册表，用于 UI/管理）

### 放什么
- 技能定义与说明（`SKILL.md`）
- 技能附带脚本（通常是 skill 自己的执行逻辑，不一定能被主程序 import）
- 任何“以技能为单位”的资源文件（如果你未来加模板/提示词/样例数据等）

### 重点
- **技能是资产**：关注的是“可发现、可索引、可运行/可引导”，而不是 Python 包结构
- **注册表是关键枢纽**：通过 `scripts/sync_skills.py` 生成 `skills/registry.json`，让 UI 能快速列出技能、提示问题、发现脚本

---

## 5) `src/services/`：服务层（业务编排/对接外部系统）

### 定位
`src/services/` 用来放“有边界的服务对象/流程编排”，通常具备：
- 业务过程（而不只是纯函数）
- 与外部系统交互（MCP、httpx、文件读写、缓存、subprocess 等）
- 状态与缓存（比如单例、TTL 缓存、连接复用）

### 放什么
- “完成一个任务链路”的模块：如 MCP 客户端服务、文件扩充服务、笔记导出服务等
- 往往会调用 `src/tools/*` 与 `src/utils/*` 做底层支撑

### 重点
`services` 更偏“流程与集成”，你在这里要优先关注：
- 稳定性（重试、超时、fallback、缓存）
- 边界（输入输出结构、错误信息可诊断）
- 可替换性（未来换实现/换供应商时影响最小）

---

## 推荐心智模型：谁负责“什么”，谁依赖“谁”

### 一句话总结
- **`scripts/`**：怎么跑（入口）
- **`src/tools/`**：可复用能力模块（运行时工具）
- **`src/utils/`**：通用辅助模块（纯函数/小封装）
- **`skills/`**：可管理的技能资产（内容/元数据/脚本/索引）
- **`src/services/`**：业务流程与外部集成（编排与边界）

### 依赖方向建议（从上到下）
`scripts/`  
→ `src/services/`（流程编排/集成）  
→ `src/tools/` / `src/utils/`（能力与通用辅助）  
并行存在一个“资产域” `skills/`（由脚本/服务去读写索引与内容）

---

## 常见决策：某个新文件该放哪

- 你要一个“命令行一键跑”的东西：放 `scripts/`
- 你要一个“很多地方会 import 的能力封装”：优先 `src/tools/`（轻量就 `src/utils/`）
- 你要一个“对接外部系统 + 缓存/重试/流程”的模块：放 `src/services/`
- 你要新增/维护一个技能：放 `skills/<skill_name>/...`，并更新/生成 `skills/registry.json`
