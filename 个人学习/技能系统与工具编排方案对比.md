## 背景与目标
你当前项目以 **LangChain** 为主干（对话 / RAG / 本地 tools），希望：
- **skills/tools 主要在本地开发**（Python 代码可调试、可测试、可版本化）
- **MCP 仅做简单体验**（例如 Streamlit UI 的天气工具）
- 后续再做“架构分层 + 动态工具注册 + 工具过滤”

因此这里把你提到的 4 条路线放在同一张地图里对比：
- **A. DeepAgent Skills（Deep Agents 技能系统）**
- **B. skillkit（技能包管理/分发/转换）**
- **C. 动态工具过滤（Middleware / 运行时工具白名单）**
- **D. LangChain MCP Adapters（把 MCP Server 工具转成 LangChain Tools）**

## A. DeepAgent Skills（Deep Agents 技能系统）
### 1) 是什么
DeepAgent Skills 是一种“把能力封装成可按需加载的技能包”的机制，通常以一个目录表示：
- `skills/<skill-name>/SKILL.md`（含 YAML frontmatter + 详细指令）
- 可选脚本/参考文档等（脚本用于确定性执行，文档用于参考）

核心理念是 **渐进式披露（Progressive Disclosure）**：
- 启动时只加载技能“简介”（name/description）
- 真正需要时再加载 `SKILL.md` 详细内容
- 再进一步按需读取脚本/参考材料

### 2) 适合你的点
- **对“个人工作台”这种多能力集合非常友好**：不用把所有细节都塞进系统提示词
- **天然利于团队沉淀**：技能目录可版本化、可审计、可复用
- **与“本地开发 tools”不冲突**：技能更偏“操作流程/策略/指南”，工具仍是代码

### 3) 典型落地方式
- 你把“写周报/整理会议纪要/PRD 生成/项目复盘”这类流程沉淀成 skill
- 由 agent 根据 skill 描述路由到合适技能，再结合本地 tools 执行

### 4) 主要代价/风险
- **需要你定义技能边界与规范**（命名、模板、前置条件、输出格式）
- **技能执行的安全性**：来自第三方的 skill 要审计（指令注入/数据泄漏）

### 5) 参考
- LangChain Blog（Deep Agents Skills）：`https://blog.langchain.com/using-skills-with-deep-agents/`

## B. skillkit（技能包管理/分发/转换）
### 1) 是什么
skillkit 更像一个 **技能生态工具链**：
- 发现/安装/同步 skills
- 在不同 agent 平台间转换技能格式（例如 `SKILL.md`、Cursor 的规则/其他 agent 的格式）
- 可选：提供服务端（REST / MCP server）做“按需技能发现”

### 2) 适合你的点
- 如果你未来会在 **多个 Agent/IDE/平台**之间迁移（Cursor、Claude Code 等），skillkit 能降低重复建设
- 适合把“技能”做成“可分发资产”，在不同项目/机器复用

### 3) 主要代价/风险
- 你当前是单项目、单 Agent 主干（LangChain + Streamlit），引入 skillkit 可能会：
  - 增加一层“技能管理系统”的复杂度
  - 让技能治理（版本/权限/来源）变成额外工作

### 4) 参考
- skillkit GitHub：`https://github.com/rohitg00/skillkit`
- PyPI：`https://pypi.org/project/skillkit/`

## C. 动态工具过滤（Middleware / 运行时工具白名单）
### 1) 是什么
核心不是“如何定义技能”，而是“在一个 agent run 的不同阶段，让模型只看到它该看到的工具”。

常见动机：
- 工具太多 → 误用概率上升、token 增长、成本增加
- 某些工具有风险（文件删除、网络请求、数据库写入）→ 需要在特定阶段才允许

### 2) 常见实现形态
- **按状态机/阶段过滤**：例如「资料入库阶段」只允许 ingest 工具；「问答阶段」才允许检索与总结工具
- **按用户权限过滤**：不同用户/模式下工具集合不同
- **按会话上下文过滤**：如果用户没上传资料，就不暴露“检索知识库”工具

### 3) 适合你的点
你已经计划：
- MCP 工具动态生成并注册到 Agent 工具字典
- 缓存 list_tools 结果，减少调用

那“动态过滤”将成为你体系里非常关键的**安全阀**与**性能阀**：
- 保证 Agent 在 UI 层“只看到必要工具”
- 在你未来工具爆炸时仍保持可控

### 4) 主要代价/风险
- 需要你定义 **可观测的状态/阶段**（否则过滤策略很难稳定）
- 需要持续维护工具元数据（风险等级、阶段、前置条件）

### 5) 参考
- 社区讨论与中间件实践（可作为思路，不要完全照搬）：`https://forum.langchain.com/t/are-dynamic-tool-lists-allowed-when-using-create-agent/1920`

## D. LangChain MCP Adapters（把 MCP Server 工具转成 LangChain Tools）
### 1) 是什么
LangChain 官方有 `langchain-mcp-adapters`，用来把 MCP server 暴露的工具（tools/resources/prompts）映射成 LangChain 可直接调用的工具对象。

你可以：
- 本地起一个 MCP Server（stdio）
- 通过 adapter 加载 server 的工具列表
- 把这些工具塞进 LangChain/LangGraph agent 的工具集合

### 2) 适合你的点
这条路线和你当前目标高度吻合：
- **MCP 仅做“简单体验”**：比如天气工具
- 工具侧保持“插件化”：未来可接更多 MCP server（也可不接）

### 3) 你需要关注的工程细节（与你的需求对应）
- **连接管理（单例）**：避免每次 UI 交互都重启 stdio server
- **工具列表缓存**：减少 `list_tools` 调用（你已提出）
- **async → sync 安全桥**：MCP SDK 异步，Streamlit 同步（你已提出）

### 4) 参考
- LangChain MCP Adapters 文档：`https://reference.langchain.com/python/langchain-mcp-adapters`

## 怎么选（结合你当前策略的建议）
### 如果你的近期目标是“先跑通分层架构 + MCP 天气 demo”
优先顺序建议：
1) **D（LangChain MCP Adapters）**：用于 MCP demo 工具快速接入
2) **C（动态工具过滤）**：用于工具爆炸前的可控性与安全
3) **A（DeepAgent Skills）**：用于把工作台业务流程沉淀成技能（中长期收益最大）
4) **B（skillkit）**：当你开始跨平台/多 Agent 分发 skills 时再上

### 一个很实用的落地组合
- 工具层：本地 tools + MCP tools（通过 adapters/或你自己的 client 封装）
- skills 层：用 DeepAgent Skills 管“流程与策略”
- middleware：做动态工具过滤（基于阶段/权限/上下文）
